# Safety, Privacy & Data Practices

> This document is a formal statement of how Argus handles data, models, and
> clinical risk — prepared for hackathon judges and any downstream integrator.

## Data

### What Argus uses
- **Only synthetic patient data** — generated by [Synthea™ (MITRE, Apache 2.0)](https://github.com/synthetichealth/synthea).
- **Public reference data** — RxNorm / RxNav (NLM), DailyMed / FDA labels, KDIGO 2022 guidelines, AGS Beers 2023, CredibleMeds KnownRisk.
- **No real Protected Health Information (PHI)** is ever used — in training, testing, demonstration, or code.

### What Argus stores
- Ephemeral per-request data during tool execution; discarded when the response is returned.
- Persistent caches: RxNorm normalization (drug reference only) and the compiled reference KB.
- Structured logs with **hashed** patient IDs only — names, DOBs, addresses, and other identifiers are never logged.

### What Argus never does
- Reads patient data from anywhere other than the SHARP-authenticated FHIR server for the current request.
- Sends patient-identifying fields to any third-party service (RxNav, Gemini).
- Persists patient data across requests.
- Writes any resources back to the FHIR server.

## Security

- **Authentication**: the MCP server accepts FHIR bearer tokens via SHARP context. It performs no independent patient-level authentication — delegation to the upstream platform is intentional.
- **Token handling**: bearer tokens are held in request-scoped `SharpContext` objects and never serialized into logs, error messages, or cache entries.
- **Transport**: HTTPS everywhere in production. Fly.io config enforces `force_https = true`.
- **No secrets in code**: all credentials via environment variables; `.env` is gitignored; `.env.example` is shipped.

## Clinician-in-the-loop

Every tool response carries the same disclaimer text:

> *"AI-generated clinical decision support. All outputs require review by a
> licensed clinician before clinical action. Built on synthetic data; not a
> substitute for clinical judgment."*

Argus is built for **decision support**, not decision-making. Concretely:
- No tool produces an auto-actionable prescription, hold, or modification.
- The `generate_med_rec_note` output explicitly marks "URGENT ACTIONS" for clinician attention rather than executing anything.
- The `reconcile_home_vs_hospital` tool reports discrepancies with confidence scores; intentionality decisions remain with the clinician.
- Every citation marker lets a reviewer trace any claim back to a specific FHIR resource.

## Model transparency

- The DDI contextual severity model is an XGBoost regressor trained on **synthetic labels**. This is disclosed in tool output via the `base_severity_source` field and in this README.
- A real production deployment would retrain on real clinical outcome labels (ADE registries, phenotyped readmissions). That work is out of scope for a hackathon.
- When the model is absent, tools fall back to a **transparent heuristic** that is deterministic and fully auditable. The fallback is advertised in logs.

## Regulatory posture

Argus does not claim regulatory clearance. The system is a **prototype for research and demonstration**. A production deployment in a clinical environment would require, at minimum:
- An institutional HIPAA Business Associate Agreement (if in the US)
- IRB review for any real-data evaluation
- Clinical validation studies (sensitivity/specificity against clinician-adjudicated gold standards)
- FDA 510(k) clearance if marketed for diagnostic/therapeutic use (CDS Hooks-style advisory scope may be exempt under the 21st Century Cures Act CDS exemption, but that determination is context-specific)
- SOC 2 / HITRUST controls for hosting infrastructure

## Responsible disclosure

Security issues and clinical safety concerns can be reported via the project's GitHub issues with the `security` or `clinical-safety` label. Please do not include PHI in bug reports.
